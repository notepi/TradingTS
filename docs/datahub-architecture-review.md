## 一、项目背景

### 1.1 项目简介

TradingAgents 是一个多智能体 LLM 金融交易框架，核心功能：

- **多智能体协作**：市场分析师、基本面分析师、新闻分析师等多角色协同
- **数据驱动决策**：从多个数据源获取股票数据、技术指标、财务报表、新闻等
- **LangGraph 执行**：基于 LangGraph 构建智能体协作流程
- **灵活配置**：支持多种 LLM 和多数据源（Tushare、YFinance）

### 1.2 触发问题的事件

**2026-04-16 发现的数据解读错误**：

在分析铂力特（688333.SH）基本面时，LLM 智能体误读了资产负债表数据。

**根本原因**：数据源返回原始数据，LLM 解读时没有字段说明。

**解决方案**：在数据源层返回值内联字段说明（见 datahub-design.md 第四章）。

## 二、架构演进历史

### 2.1 第一阶段：七层架构（原始）

```
┌─────────────────────────────────────────────────────────────────┐
│ 7层架构（过于复杂）                                              │
├─────────────────────────────────────────────────────────────────┤
│ 1. 实现层 - datasource/{vendor}/*.py                            │
│ 2. 接口层 - datasource/{vendor}/__init__.py                     │
│ 3. 抽象层 - datasource/tools/interface.py                       │
│ 4. 声明层 - datasource/tools/tools_registry.yaml                │
│ 5. 决策层 - tradingagents/config.yaml                           │
│ 6. 工具层 - datasource/tools/*_tools.py                         │
│ 7. 智能体层 - tradingagents/config/agents_registry.yaml         │
└─────────────────────────────────────────────────────────────────┘
```

**问题**：
| 问题 | 描述 |
|------|------|
| interface.py 跨三层 | 读多个配置文件，依赖混乱 |
| description 冗余 | 多处重复定义工具说明 |
| 层级过细 | 边界模糊，难以理解 |

### 2.2 第二阶段：三层架构（含 tools 层）

```
┌─────────────────────────────────────────────────────────────────┐
│ 3. 智能体层 - tradingagents/                                     │
├─────────────────────────────────────────────────────────────────┤
│ 2. 数仓层 - datasource/datahub/                                  │
│    ├─ servers/ - @tool 定义 + 服务接口                           │
│    └─ tools/ - vendor 路由 + API 调用                            │
├─────────────────────────────────────────────────────────────────┤
│ 1. 数据源层 - datasource/{vendor}/                              │
└─────────────────────────────────────────────────────────────────┘
```

**问题**：
| 问题 | 描述 |
|------|------|
| tools 层职责不清 | 只是转发层，价值不明显 |
| servers/tools 边界模糊 | 职责重叠 |

### 2.3 第三阶段：两层架构（最终）

```
┌─────────────────────────────────────────────────────────────────┐
│ 3. 智能体层 - tradingagents/                                     │
│    职责：智能体实现 + LLM配置 + 工具绑定                          │
├─────────────────────────────────────────────────────────────────┤
│ 2. 数仓层 - datasource/datahub/servers/                         │
│    职责：@tool 定义 + vendor 路由                                │
├─────────────────────────────────────────────────────────────────┤
│ 1. 数据源层 - datasource/{vendor}/                              │
│    职责：API 实现 + 字段说明                                     │
└─────────────────────────────────────────────────────────────────┘
```

**决策**：去掉 tools 层，简化为两层。

## 三、关键设计讨论

### 3.1 讨论：是否保留 tools 层？

| 方案 | 职责 | 价值 | 决策 |
|------|------|------|------|
| 保留 tools 层 | vendor 路由 + API 调用 | 只是转发层 | ❌ 放弃 |
| 删除 tools 层 | servers 直接调用数据源 | 简化架构 | ✓ 采用 |

**决策理由**：
1. tools 层实际只是转发，无独立价值
2. vendor 路由可放在 servers 层（读配置）
3. 减少中间层，流程更直接

### 3.2 讨论：vendor 路由放哪里？

| 方案 | 位置 | 决策 |
|------|------|------|
| 函数层硬编码 | @tool 参数里写 vendor | ❌ 放弃 |
| 类别级路由 | registry.yaml 按类别路由 | ❌ 放弃 |
| 配置统一管理 | registry.yaml（default_vendor + tool_vendors） | ✓ 采用 |

**决策理由**：
1. 函数层不应关心 vendor，只写业务逻辑
2. 配置层统一管理，一目了然
3. 新函数自动用 default_vendor

### 3.3 讨论：字段说明放哪里？

| 方案 | 位置 | 决策 |
|------|------|------|
| tools_registry.yaml | 配置文件 | ❌ 放弃 |
| servers 层预处理 | 数仓层 | ❌ 放弃 |
| 数据源层返回值内联 | datasource/{vendor}/data.py | ✓ 采用 |

**决策理由**：
1. 数据源层知道字段含义
2. 返回值内联，LLM 直接可读
3. 不增加额外处理层

### 3.4 讨论：工具描述放哪里？

| 方案 | 位置 | 决策 |
|------|------|------|
| tools_registry.yaml | 配置文件 | ❌ 放弃 |
| @tool docstring | servers/{domain}.py | ✓ 采用 |

**决策理由**：
1. docstring 是 LangChain 标准做法
2. 和代码在一起，易于维护
3. 减少配置文件内容

### 3.5 讨论：函数签名如何约定？

| 方案 | 约定 | 决策 |
|------|------|------|
| 参数映射 | registry.yaml 配置映射 | ❌ 放弃 |
| 强制一致 | 数据源层定义，找不到报错 | ✓ 采用 |

**决策理由**：
1. 避免参数映射增加复杂度
2. 数据源层是基准，数仓层适配
3. 历史遗留函数用别名或包装适配

### 3.6 讨论：如何简化注册机制？

| 方案 | 做法 | 决策 |
|------|------|------|
| 手动维护 | 在 data.py 维护导出列表 | ❌ 放弃 |
| @export 装饰器 | 自动扫描注册 | ✓ 采用 |

**决策理由**：
1. 写函数加一行 @export 即可
2. 自动扫描，避免配置和代码不一致

## 四、设计决策记录

### D1: 两层架构（不含 tools 层）

**决策**：智能体 → servers → 数据源

**理由**：
1. tools 层只是转发层，价值不明显
2. servers 可直接调用数据源
3. 减少中间层，架构更清晰

**影响**：servers 层职责：@tool + vendor 路由

### D2: vendor 路由配置统一管理

**决策**：registry.yaml 存 default_vendor + tool_vendors

**理由**：
1. 函数层不关心 vendor
2. 配置统一，一目了然
3. 换数据源只改配置

**影响**：@tool 函数不写 vendor 参数

### D3: 字段说明在数据源层

**决策**：返回值内联字段说明

**理由**：
1. 数据源层知道字段含义
2. LLM 直接可读
3. 不增加额外处理

**影响**：datasource/{vendor}/data.py 返回字符串（含字段说明）

### D4: 工具描述在 @tool docstring

**决策**：docstring 描述工具用途

**理由**：
1. LangChain 标准做法
2. 和代码在一起
3. 减少配置内容

**影响**：不在 tools_registry.yaml 写 description

### D5: 函数签名必须一致

**决策**：数据源层定义，找不到报错

**理由**：
1. 避免参数映射复杂度
2. 强制约定

**影响**：历史遗留函数用别名适配

### D6: @export 装饰器自动注册

**决策**：自动扫描注册

**理由**：
1. 写函数加一行
2. 避免手动维护

**影响**：data.py 自动收集导出

## 六、架构对比

| 维度 | 七层架构 | 三层架构（含 tools） | 两层架构（最终） |
|------|----------|---------------------|-----------------|
| 层数 | 7 | 3（数仓含 2 子层） | 2 |
| 注册机制 | 3个 | 2个 | 1个 |
| vendor 路由 | 多处 | tools 层 | servers 层 |
| 字段说明 | 无 | 未定 | 数据源层 |
| 工具描述 | 配置文件 | 配置文件 | @tool docstring |
| 加新函数改动 | 改多处 | 改多处 | 三层各一处 |
